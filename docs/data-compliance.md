# แบบออกแบบระบบข้อมูลและ Compliance — แพลตฟอร์มไกด์ฮาลาลประเทศไทย

## 1. Data Pipeline: Bootstrap ฐานข้อมูล

### 1.1 สถาปัตยกรรม 3 ชั้น (สำคัญที่สุดเรื่อง ODbL)

```
ชั้น 1: raw_source_records   — ข้อมูลดิบต่อแหล่ง แยกตาราง/แยก schema ตามแหล่ง (immutable, เก็บ license ต่อ record)
ชั้น 2: match_candidates     — ผลการจับคู่ข้าม source + คะแนน + สถานะ (auto/needs_human/rejected)
ชั้น 3: places (canonical)   — listing จริงที่แสดงผล สร้างจาก "ข้อเท็จจริงที่ทีมยืนยันเอง" + provenance ต่อ field
```

**หลักการจัดการ ODbL ของ OSM:** ใช้ข้อมูล OSM เป็น **"lead list" (รายชื่อนำทางให้ทีมไปตรวจ)** ไม่ copy เข้า canonical โดยตรง — พิกัดและชื่อใน canonical ต้องมาจากการยืนยันของทีมเอง (ปักหมุด ณ สถานที่จริง / เทียบทะเบียนราชการ / เจ้าของยืนยัน) จึงถือว่า canonical DB เป็นฐานข้อมูลที่เราสร้างเอง ไม่ใช่ derivative database ของ OSM:
- ตาราง `raw_osm_places` แยกเด็ดขาด มี field `osm_id`, `osm_version`, `license='ODbL'`
- record ใน canonical ที่ "ยังไม่ยืนยัน" และแสดงผลโดยอิงพิกัด OSM ตรงๆ → ติด flag `odbl_derived=true` และแสดง attribution "© OpenStreetMap contributors" บนหน้านั้น + หน้า `/about/data-sources` รวม attribution ทุกแหล่ง (OSM, GD Catalog ใต้ Open Government Data License, TAT, CICOT)
- แผนที่ (OpenFreeMap tiles) ต้องมี attribution OSM มุมแผนที่เสมอ — อันนี้บังคับอยู่แล้ว
- ถ้าอนาคตอยาก export ข้อมูลชั้นที่ยัง odbl_derived → ต้องปล่อยเป็น ODbL; ข้อมูลชั้นยืนยันเองแล้ว → license ของเรา

### 1.2 ลำดับการ ingest (4 สเตจ, รันซ้ำได้แบบ idempotent)

| สเตจ | แหล่ง | ได้อะไร | วิธีเข้าถึง | หมายเหตุ |
|---|---|---|---|---|
| A | **OSM Overpass** | มัสยิด/ที่ละหมาด **พร้อมพิกัด** + name:th/name:en | Query `nwr["amenity"="place_of_worship"]["religion"="muslim"]` bbox ประเทศไทย, ฟรี | รันก่อนเพราะเป็นแหล่งเดียวที่มีพิกัด — เป็นแกน geo ให้แหล่งอื่นมา match |
| B | **GD Catalog (กรมการศาสนา, ชุด dra-04-02)** | ทะเบียนมัสยิดทางการ ~4,059 แห่ง (ชื่อทางการ, เลขทะเบียน, ตำบล/อำเภอ/จังหวัด) **ไม่มีพิกัด** | CSV/XLSX ดาวน์โหลด | เป็น "source of truth ด้านชื่อ+ตัวตน" → match เข้ากับพิกัด OSM |
| C | **TAT Data API + Thailand Tourism Directory Open API** | สถานที่ท่องเที่ยว/ร้าน/ที่พัก 7 เมือง MVP (มีพิกัด, หลายภาษา บางส่วน) | ลงทะเบียน API key, อ่าน ToS เรื่อง attribution/การเก็บ cache ก่อนใช้ | กรองเฉพาะ 7 จังหวัด MVP, tag `source='tat'` |
| D | **CICOT/THIC** | ร้าน/สถานประกอบการรับรองฮาลาล (~3,500 ร้าน), เลขที่ใบรับรอง | **ไม่มี API** — แนะนำทำหนังสือขอความร่วมมือ/MOU เป็นทางการก่อน; ระหว่างรอ ให้ทีม admin คีย์มือจากระบบค้นหา halal.or.th เฉพาะเมือง MVP | ข้อมูลนี้คือหัวใจของ trust level L1 — ห้าม scrape แบบเสี่ยงผิดใจหน่วยงานที่เราต้องเป็นพันธมิตรระยะยาว |

หลัง 4 สเตจ → เปิดช่องทางเฟส 3-4 (owner claim, user submission) ซึ่งเขียนเข้า `places` ผ่าน verification workflow ในข้อ 2 เท่านั้น

### 1.3 Dedupe / Entity matching ข้ามแหล่ง (ปัญหาชื่อไทยสะกดต่างกัน)

Pipeline จับคู่แบบ score-based สองชั้น:

**ชั้น blocking (ลดคู่เทียบ):** จับคู่เฉพาะ record ที่อยู่ (ก) ตำบล/อำเภอเดียวกัน (map จากที่อยู่ GD Catalog → รหัส อปท.) หรือ (ข) ห่างกัน ≤ 500 ม. ถ้ามีพิกัดทั้งคู่

**ชั้น scoring:**
1. **Normalize ชื่อไทยก่อนเทียบ:** ตัดคำนำหน้า (มัสยิด/สุเหร่า/บาลาย/มูศ็อลลา), ตัดวรรณยุกต์+การันต์+ไม้ไต่คู้ (นูรุ้ลอิสลาม = นูรุลอิสลาม = นูรุลอิสลาม), ยุบสระซ้ำ, ตัดช่องว่าง/ขีด — ชื่อมัสยิดไทยส่วนใหญ่ทับศัพท์อาหรับจึงสะกดแกว่งที่วรรณยุกต์และตัวสะกดท้ายพยางค์เป็นหลัก
2. คะแนน = trigram similarity ของชื่อ normalized (ใช้ `pg_trgm`) × 0.5 + ความใกล้เชิงพื้นที่ × 0.3 + ที่อยู่ระดับตำบลตรงกัน × 0.2
3. เกณฑ์: ≥ 0.85 → auto-merge (บันทึก match ไว้ย้อนดูได้), 0.55–0.85 → เข้า **human review queue ใน admin** (แสดงคู่เทียบ side-by-side + แผนที่), < 0.55 → คนละแห่ง
4. ห้าม merge แบบทำลายข้อมูล — canonical เก็บ `source_refs[]` ชี้กลับ raw ทุกตัว และเก็บ provenance ต่อ field (`name.source`, `location.source`) เพื่อรู้ว่า field ไหนติด ODbL

## 2. Verification Workflow

### 2.1 สถานะ (state machine ระดับ listing)

```
draft → pending_review → verified ⇄ needs_reverify → archived
                ↘ rejected
สถานะเสริม (flag ซ้อนได้): disputed (ถูกรายงานว่าข้อมูลฮาลาลผิด อยู่ระหว่างสอบ)
```

- `draft` — ingest อัตโนมัติ/ผู้ใช้ส่งเข้า ยังไม่แสดงสาธารณะ
- `pending_review` — ข้อมูลครบขั้นต่ำ (ชื่อ, พิกัด, หมวด, เมือง) รอ moderator; **แสดงสาธารณะได้เฉพาะหมวดมัสยิดจากทะเบียนราชการ** (ความเสี่ยงต่ำ, ติดป้าย "ข้อมูลจากทะเบียนทางการ ยังไม่ตรวจภาคสนาม")
- `verified` — ผ่านการยืนยันตามเกณฑ์ trust level แสดงเต็มรูปแบบ
- `needs_reverify` — trigger อัตโนมัติ: ใบรับรองหมดอายุ / เกินรอบ re-verify / มีรายงาน 3 ครั้งขึ้นไป / เจ้าของแจ้งเปลี่ยนมือ → ยังแสดงได้แต่ badge ถูก downgrade
- `archived` — ปิดกิจการ/ซ้ำซ้อน (เก็บไว้ ไม่ลบ, redirect ไป listing ใหม่ถ้า merge)

### 2.2 ใครยืนยันอะไรได้ (permission matrix)

| ข้อมูล | user | เจ้าของร้าน (claimed) | field verifier | moderator | admin |
|---|---|---|---|---|---|
| เสนอแก้ข้อเท็จจริงพื้นฐาน (เวลาเปิด/โทร/เมนู) | เสนอ→queue | แก้ได้เลย (log + spot check) | ✔ | ✔ | ✔ |
| พิกัด/ที่อยู่ | เสนอ | เสนอ | ✔ (ปักหมุดหน้างาน) | ✔ | ✔ |
| Trust L3 (no pork/no alcohol) | — | เสนอ+หลักฐาน | ✔ | ✔ | ✔ |
| Trust L2 (Muslim-owned) | — | เสนอ+หลักฐาน | ✔ (พบ/โทรคุยเจ้าของ) | — | ✔ |
| **Trust L1 (รับรอง กอท.)** | — | อัปโหลดใบรับรอง | เก็บหลักฐานหน้างาน | — | **admin เท่านั้น** (ตรวจเลขที่กับฐาน CICOT ก่อนอนุมัติ) |
| เปลี่ยนสถานะ listing | — | — | — | ✔ (ยกเว้น L1) | ✔ |

### 2.3 รอบ re-verify + field บังคับ

- ทุก listing มี `last_verified_at`, `verified_by` (user id + role), `verification_method` (`site_visit | phone | document | official_registry | owner_attestation`), `evidence[]` (ไฟล์แนบภายใน)
- ใบรับรอง กอท.: เก็บ `cert_number`, `cert_expires_at` → cron แจ้งเตือนที่ 60/30/7 วันก่อนหมด (แจ้งทั้ง admin และเจ้าของที่ claim) → **วันที่หมดอายุ ระบบ downgrade อัตโนมัติ** เป็น "ใบรับรองหมดอายุ – รอยืนยันการต่ออายุ" (ไม่ถอด listing แต่ถอด badge L1)
- รอบตามเวลา: L1 ทุก 12 เดือน (ตามรอบใบรับรองอยู่แล้ว), L2/L3 ทุก 6 เดือน (โทรเช็คพอ), มัสยิด/ห้องละหมาด ทุก 24 เดือน, listing ที่มี traffic สูงสุด 10% แรกของเมือง → เช็คถี่ขึ้นเท่าตัว
- ทุกหน้า listing แสดง "ตรวจสอบล่าสุด: [วันที่] โดย [ทีมงาน/เอกสารทางการ]" — เป็นทั้ง trust signal และ SEO freshness

## 3. Halal Trust Taxonomy (ฉบับใช้งานจริง + ปลอดภัยตาม ป.อาญา ม.272-273)

**หลักเหล็ก:** เราไม่เคย "รับรองฮาลาล" เอง และ**ไม่แสดงตราฮาลาลของ กอท. (โลโก้/เครื่องหมายรับรอง) ในระบบจนกว่าจะมีข้อตกลงเป็นลายลักษณ์อักษร** — badge ทุกอันเป็นดีไซน์ของแพลตฟอร์มที่สื่อสาร "แหล่งที่มาของการยืนยัน" ไม่ใช่การรับรอง

| ระดับ | ชื่อ badge (ของเรา) | หลักฐานขั้นต่ำ | ข้อความประกอบที่แสดง |
|---|---|---|---|
| **L1** | "มีใบรับรองฮาลาล กอท." | เลขที่ใบรับรอง + วันหมดอายุ + รูปถ่ายใบรับรอง + admin ตรวจไขว้กับระบบค้นหาของ CICOT | "ยืนยันจากใบรับรองเลขที่ … หมดอายุ … ตรวจสอบล่าสุด [วันที่] • ข้อมูลการรับรองเป็นของ สนง.คณะกรรมการกลางอิสลามแห่งประเทศไทย" |
| **L2** | "เจ้าของเป็นมุสลิม" | ทีมพบ/โทรคุยกับเจ้าของ + คำยืนยันของเจ้าของ (บันทึกไว้) + ไม่พบหมู/แอลกอฮอล์ในเมนู | "ข้อมูลจากการยืนยันของเจ้าของร้านและการตรวจของทีมงาน — ร้านนี้ไม่มีใบรับรองฮาลาลอย่างเป็นทางการ" |
| **L3** | "Muslim-friendly (ไม่มีหมู/ไม่เสิร์ฟแอลกอฮอล์)" | ตรวจเมนู (ภาคสนามหรือรูปถ่าย) + ข้อมูลจากร้าน | "ตรวจจากเมนูและข้อมูลของร้าน — โปรดสอบถามร้านเพิ่มเติมหากท่านมีข้อกำหนดเคร่งครัด" |
| **L4** | "ข้อมูลจากชุมชน (ยังไม่ยืนยัน)" | มีผู้ใช้ส่งข้อมูล/พบใน OSM แต่ทีมยังไม่ตรวจ | "ยังไม่ผ่านการตรวจสอบโดยทีมงาน" (badge สีเทา) |

Attribute เสริมที่เป็น filter แยกจากระดับ (สไตล์ Halal Navi): เสิร์ฟแอลกอฮอล์ในร้าน (โรงแรม), แยกครัว/ภาชนะ, มีห้องละหมาด, มีที่อาบน้ำละหมาด, เนื้อ zabihah

**กรณีพิเศษ:**
- **ใบรับรองหมดอายุ:** เปลี่ยน badge เป็น "ใบรับรองหมดอายุ – รอการต่ออายุ" (สีเหลือง) ไม่ลบประวัติ; ถ้าเกิน 90 วันไม่มีหลักฐานต่ออายุ → ตกไป L2/L3 ตามหลักฐานที่เหลือ
- **ร้านเปลี่ยนเจ้าของ:** ใบรับรอง กอท. ผูกกับผู้ประกอบการ ไม่ใช่ทำเล → รายงาน "เปลี่ยนเจ้าของ" trigger `needs_reverify` ทันที + ถอด L1/L2 ชั่วคราวจนกว่าจะตรวจใหม่
- **ถูกโต้แย้ง (disputed):** ระหว่างสอบสวนแสดง "ข้อมูลอยู่ระหว่างการตรวจสอบซ้ำ" — **ห้าม**แสดงข้อความว่า "ถูกรายงานว่าไม่ฮาลาล" (เสี่ยงหมิ่นประมาทโดยแพลตฟอร์มเอง)
- Disclaimer กลางท้ายทุกหน้า listing: "สถานะฮาลาลจัดทำโดย [ชื่อแพลตฟอร์ม] จากหลักฐานที่ระบุไว้ ไม่ใช่การรับรองของหน่วยงานศาสนา ผู้ใช้ควรใช้วิจารณญาณประกอบ"

## 4. ระบบรีวิวที่ปลอดภัยตามกฎหมายไทย

### 4.1 หลักออกแบบ
หมิ่นประมาทเป็น **คดีอาญา** (ม.326/328 + พ.ร.บ.คอมฯ ม.14(1)) และแพลตฟอร์มเสี่ยง ม.15 → แยกเนื้อหาเป็น 2 ท่อเด็ดขาด:

**ท่อ 1 — รีวิวสาธารณะ:** ประสบการณ์การใช้บริการ (รสชาติ อาหาร บริการ ความสะอาดเชิงประสบการณ์ ราคา) เท่านั้น
**ท่อ 2 — "รายงานข้อมูลฮาลาลไม่ถูกต้อง" (ส่วนตัว):** ทุกข้อกล่าวหาเชิงข้อเท็จจริงร้ายแรง (พบหมู/แอลกอฮอล์, ใบรับรองปลอม/หมดอายุ, ไม่ใช่มุสลิมแล้วอ้าง) → form ส่งเข้า admin queue เงียบๆ **ไม่โพสต์สาธารณะเด็ดขาด** ผู้รายงานแนบรูปได้ เลือกให้ติดต่อกลับได้ ผลการสอบสะท้อนออกมาเป็น "การเปลี่ยน trust status โดยทีมงาน" ไม่ใช่คำประจาน — วิธีนี้ทั้งปกป้องผู้ใช้จากการถูกฟ้อง และปกป้องแพลตฟอร์ม

### 4.2 Moderation ต่อประเภทเนื้อหา

| เนื้อหา | โหมด |
|---|---|
| คะแนนดาวอย่างเดียว | auto-publish |
| ข้อความรีวิวทั่วไป | post-moderation (publish ทันที + keyword screen + ปุ่ม report) |
| ข้อความที่ชน keyword เสี่ยง (ไม่ฮาลาล, หลอก, โกง, ปลอม, สกปรกจนป่วย, ชื่อบุคคล) | **pre-moderation** — hold ไว้ให้ moderator อ่านก่อน ภายใน 24 ชม. → อนุมัติ/ขอแก้/แปลงเป็นรายงานท่อ 2 |

Review guidelines (แสดงก่อนกดส่งทุกครั้ง ย่อ 4 ข้อ): เล่าจากประสบการณ์ตรง • ไม่กล่าวหาเรื่องผิดกฎหมาย/ความไม่ฮาลาล — ใช้ปุ่ม "รายงานข้อมูลฮาลาล" แทน • ไม่ระบุชื่อ-รูปบุคคล • ผู้รีวิวรับผิดชอบเนื้อหาตามกฎหมายไทยซึ่งหมิ่นประมาทเป็นความผิดอาญา

### 4.3 Notice-and-Takedown ตามประกาศ MDES (24 ชม.)

- **ช่องทาง:** หน้า `/legal/takedown` (5 ภาษา) + อีเมล legal@ — form บังคับ: ตัวตนผู้ร้อง, URL เนื้อหา, เหตุผล/ข้อกฎหมาย, คำยืนยันความจริง
- **SLA:** ยืนยันรับเรื่องอัตโนมัติทันที → moderator ประเมินใน 4 ชม. ทำการ → **ซ่อนเนื้อหาชั่วคราว (suspend ไม่ใช่ลบ) ภายใน 24 ชม. นับจากรับคำร้องเสมอ** ถ้ายังตัดสินไม่ได้ให้ซ่อนก่อน — ต้นทุนของการซ่อนผิดต่ำกว่าค่าปรับ 200,000 + วันละ 5,000 มาก
- **Log ทุกขั้น** ในตาราง `takedown_requests`: received_at, acknowledged_at, actioned_at, action, ผู้ดำเนินการ, เหตุผล — เก็บ ≥ 2 ปี พร้อม export ให้เจ้าหน้าที่
- **Escalation:** on-call rotation (moderator หลัก → admin → ที่ปรึกษากฎหมาย) มีเบอร์โทรใน runbook; แจ้งผู้โพสต์พร้อมช่องทาง counter-notice (คืนเนื้อหาได้เมื่อทีมกฎหมายเห็นว่าปลอดภัย)
- เก็บ traffic log ตามหน้าที่ผู้ให้บริการ พ.ร.บ.คอมฯ ม.26 (90 วัน)

## 5. PDPA-first Design (ศาสนา = ข้อมูลอ่อนไหว ม.26)

### 5.1 ใช้ได้โดยไม่ต้อง login (ให้มากที่สุด — ลด exposure ทั้งหมด)
ค้นหา/แผนที่/หน้า listing ทุกหน้า, เวลาละหมาด + กิบลัต (**คำนวณฝั่ง client ด้วย adhan-js + ตารางจุฬาราชมนตรีที่ bundle ไว้ — พิกัดผู้ใช้ไม่ออกจากเครื่อง**), near-me (ส่งพิกัดขึ้น server เฉพาะ query นั้น ไม่ผูก identity, log ตัดทศนิยมพิกัดเหลือ 2-3 ตำแหน่ง), เปลี่ยนภาษา, บันทึกรายการโปรด + trip plan ใน localStorage/IndexedDB, offline city pack, ส่ง takedown/รายงานข้อมูล

**ต้อง login เฉพาะ:** เขียนรีวิว/อัปโหลดรูป, claim listing, sync รายการโปรดข้ามเครื่อง, ส่ง listing ใหม่ — สมัครด้วยอีเมล/OAuth ขอแค่ display name

### 5.2 Data minimization + consent
- **ไม่มี field ศาสนาในระบบเลย ไม่ถาม ไม่อนุมาน** — การใช้แอปไม่ถูกตีความเป็นข้อมูลศาสนาเพราะเราไม่บันทึก profile เชิงพฤติกรรมศาสนารายบุคคล; ห้ามสร้าง segment แบบ "ผู้ใช้มุสลิม" ใน analytics/marketing
- ฟีเจอร์อนาคตที่แตะข้อมูลอ่อนไหวจริง (prayer tracker, แจ้งเตือนละหมาดแบบ server-side) → **explicit consent แยก** เป็น opt-in ต่างหากจาก T&C + เก็บ consent log (`consents`: user_id, purpose, version, granted_at, revoked_at) + ถอนได้ในหน้า settings
- Analytics: **Plausible หรือ Umami (self-host, cookieless)** → ไม่ต้องมี cookie banner สำหรับ analytics; ใช้ cookie เฉพาะ session ที่จำเป็น → banner แบบเบา แจ้ง + ลิงก์นโยบาย
- Retention: ลบบัญชี → ลบ/anonymize ใน 30 วัน (รีวิวคง anonymized), รายงานฮาลาล 1 ปีหลังปิดเคส, takedown log 2 ปี, traffic log 90 วัน
- Hosting: เลือก region สิงคโปร์ (หรือไทย) และประกาศ cross-border transfer ในนโยบาย

### 5.3 Privacy policy (5 ภาษา — แปลโดยมนุษย์เท่านั้น) ต้องมี
ผู้ควบคุมข้อมูล+ช่องทางติดต่อ • รายการข้อมูล+วัตถุประสงค์+ฐานทางกฎหมายรายรายการ • คำอธิบายเรื่องข้อมูลอ่อนไหวและแนวทาง "ไม่เก็บศาสนา" • การใช้พิกัด (client-side) • retention ต่อประเภท • สิทธิเจ้าของข้อมูล (เข้าถึง/แก้/ลบ/ถอน consent/ร้องเรียน สคส.) • ผู้รับข้อมูล/ผู้ประมวลผลช่วง • cross-border • DPO/ผู้ประสานงาน • เวอร์ชัน+วันที่

## 7. KPI ตั้งแต่วันแรก

**Supply (สุขภาพฐานข้อมูล):** จำนวน listing แยก เมือง×หมวด×trust level • % verified ต่อเมือง (เป้า launch ≥ 30%, +6 เดือน ≥ 60% ของ listing มี traffic) • median "อายุข้อมูล" (วันนับจาก last_verified_at) • backlog ใบรับรองใกล้หมดอายุ (ต้อง = 0 ที่ค้างเกิน 7 วันหลังหมด)
**Demand:** organic sessions แยก locale×เมือง (ดู ms/id/ar แยกเสมอ) • อันดับ SERP ของ keyword cluster หลัก ("halal food + เมือง", "makanan halal + เมือง", "مطعم حلال بانكوك") • PWA install + offline pack downloads
**Engagement (proxy conversion):** CTR ค้นหา→หน้า listing • คลิก "นำทาง/โทร/LINE" ต่อ session • saves/trip plans ที่สร้าง
**Trust & Safety:** รายงานข้อมูลฮาลาล: จำนวน + median time-to-resolve (เป้า ≤ 72 ชม.) • takedown SLA 24 ชม. = 100% (ตัวเดียวที่ห้ามพลาด) • % รีวิว pre-mod ที่เคลียร์ใน 24 ชม.
**Ops:** ต้นทุนต่อ 1 verified listing • verifications/สัปดาห์ต่อคน
**Seasonal:** traffic ช่วงรอมฎอน/อีด/ปิดเทอมมาเลเซีย/ซัมเมอร์ GCC เทียบ baseline — ใช้วางแผนคอนเทนต์ปีถัดไป

**ลำดับการลงมือ:** (1) schema canonical + provenance → (2) ingest OSM+GD Catalog + dedupe queue → (3) admin verification workflow + trust taxonomy → (4) หน้า legal/privacy/takedown + log → (5) ingest TAT + คีย์ CICOT → (6) รีวิว 2 ท่อ (หลังมี moderation พร้อม) — ระบบรีวิวเปิดทีหลังสุดได้ ไม่บล็อก launch
